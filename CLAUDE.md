# CLAUDE.md

このプロジェクトは**仕様駆動開発（Specification-Driven Development）**で進める。

## 開発フロー

```
仕様書作成 → プランモード確認 → 実装 → ビルド・テスト
```

### 1. 仕様書作成（必須・最初のステップ）

実装前に必ず `./specs/` 以下に仕様書を作成する。

```
specs/
  <機能名>.md     # 各機能の仕様書
```

仕様書に含めるべき内容：

- **概要**: 機能の目的と背景
- **ユースケース**: アクター・事前条件・基本フロー・代替フロー・事後条件
- **データ定義**: 入出力のデータ構造
- **制約・非機能要件**: パフォーマンス、エラー処理方針など
- **受け入れ条件**: 完了の定義（テストケース含む）

### 2. プランモード（実装前確認）

仕様書作成後、実装を開始する前に **必ずプランモードに入る**。

- `EnterPlanMode` を使って実装計画を提示する
- ユーザーの承認を得てから実装を開始する
- 計画には影響ファイル・変更内容・実装順序を含める

### 3. 実装

- 言語: **Go**
- 仕様書のユースケースを単位として実装する
- 各ユースケースに対応するテストを必ず作成する
- パッケージ構成はユースケース／ドメイン境界に沿って設計する

### 4. ビルド・テスト

ビルドには **Makefile** を使用する。

```bash
make build   # ビルド
make test    # テスト実行
make lint    # リント
make run     # 実行
```

---

## 言語・ツール規約

### Go

- Go modules を使用（`go.mod` が必須）
- エラーハンドリングは明示的に行う（`errors.As` / `errors.Is` を活用）
- インターフェースはユースケース境界に定義する
- テストは実装と同時に `*_test.go` ファイルに作成する（後回し禁止）
- `testing` パッケージを使う
- テーブル駆動テストを推奨する

### Makefile

- `make help` でターゲット一覧を表示できるようにする
- 変数（`BINARY_NAME`, `GOFLAGS` など）を先頭にまとめる
- `.PHONY` を適切に設定する

## ディレクトリ構成（推奨）

```
.
├── CLAUDE.md
├── Makefile
├── go.mod
├── go.sum
├── specs/               # 仕様書（使用駆動開発の起点）
│   └── <機能名>.md
├── cmd/                 # エントリーポイント
│   └── <app>/
│       └── main.go
├── internal/            # 内部パッケージ
│   ├── usecase/         # ユースケース層
│   ├── domain/          # ドメインモデル
│   ├── infra/           # インフラ層（DB・外部API等）
│   └── handler/         # HTTP等のハンドラ層
└── pkg/                 # 外部公開パッケージ（必要な場合のみ）
```

---

## 作業ルール

1. **仕様書なしに実装を開始しない**
2. **プランモードの承認なしに実装を開始しない**
3. 仕様書と実装が乖離した場合は仕様書を先に更新する
4. `specs/` の仕様書はドキュメントとして常に最新状態を保つ
5. 過度な抽象化・将来の要件への先回りはしない（YAGNI）
6. テストは実装と同時に作成する（実装後に追加するのではなく、実装ファイルと対になる `*_test.go` を同タイミングで作成する）
7. テストなしの実装は完了とみなさない
